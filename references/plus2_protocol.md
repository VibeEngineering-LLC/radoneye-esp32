# RadonEye Plus2 (RD200PLUS) — протокол BLE (live-verified)

Документ описывает BLE-протокол прибора **RadonEye Plus2 / RD200PLUS** по данным
**прямого live-реверса 2026-06-15** (чтение через Bluetooth ПК, `bleak`/WinRT).

> **Приватность.** Реальный MAC прибора маскирован на `AA:BB:CC:DD:EE:FF`.
> Серийный номер в adv-имени маскирован: показан только префикс поколения `FR:PD`
> (дискриминатор), цифры серийника — `FR:PD<serial>`. Реальные значения — в
> приватном control-space агента, в публичный репо не попадают.

---

## 1. Идентификация прибора

| Параметр | Значение |
|---|---|
| Модель | RD200PLUS (семейство «Plus2») |
| adv local_name | `FR:PD<serial>` (префикс поколения `FR:PD`, цифры серийника маскированы) |
| Префикс поколения | **`FR:PD`** |
| MAC (live) | `AA:BB:CC:DD:EE:FF` (маска; реальный приватный) |
| Manufacturer Name String (char 0x2a29) | читается (значение не снято в этой сессии) |
| RSSI при снятии | ≈ -51 dBm (прибор рядом с ПК) |

**Классификация поколения.** Префикс adv-имени `FR:PD` **не совпадает** ни с
`FR:R2` (RD200 <2022, см. EtoTen), ни с `FR:RU` (RD200 ≥2022). По GATT-карте прибор
ближе к V1-семейству (service `1523`, write `1524`, notify `1525`/`1526`,
Nordic-base UUID `…-1212-efde-1523-785feabcd123`), **но формат ответа на 0x50 —
uint16 LE Bq/m³** (как у V2/type1, не float32 pCi/L). То есть Plus2 — **отдельное
поколение**: V1-style GATT + V2-style числовой формат.

## 2. Provenance (как снято)

- **Метод**: прямое чтение через Bluetooth ПК (`bleak` на Python 3.12, WinRT-бэкенд).
  ESP32-мост предварительно отключён от прибора (single-central link освобождён).
- **Скрипты** (в `scripts/` этого скилла):
  - `plus2_gatt_dump.py` — полный дамп GATT (service/char/descriptor) + запись
    0x50/0x51 + захват notify.
  - `plus2_poll.py` — multi-sample 0x50 с двойной интерпретацией кадра (uint16 Bq
    vs float32 pCi/L) для однозначного определения формата.
  - `plus2_0x51_probe.py` — целевой multi-sample 0x51 (окно 110 с, паузы 12 с) для
    разметки полей кадра 0x51 + проверка notify-1526 (закрытие хвостов §9).
  - `plus2_opcode_probe.py` — безопасный probe whitelist-опкодов `0x10`/`0xE8`/`0xE9`
    (поиск источника day/month-средних) + повторная проверка notify-1526. Маскирует
    MAC/serial в выводе → лог публикабелен.
  - `plus2_10e8e9_probe.py` — тщательный «снимок» `0x10`/`0xE8`/`0xE9` (маркер 0x50 +
    по 3 записи каждого опкода, окна 10 с, обе notify). Результат 2026-06-15: все три
    опкода — **0 кадров** (по 3 попытки каждый), notify-1526 молчит. Маскирует MAC/serial.
- **Дата**: 2026-06-15. Прибор: реальный (live), показания радона ≈ 18 Bq/m³
  (сессия 1) / ≈ 26 Bq/m³ (probe-2) / ≈ 34 Bq/m³ (probe-3) — естественный дрейф радона.
  Три probe-сессии — внутри одного дня, ~23–25 мин врозь (см. §6, доказано по live-часам).
- **Дополнительно (2026-06-15) — трасса официального приложения**: снят Android HCI
  Bluetooth snoop (btsnoop_hci) приложения RadonEye+² (`kr.ftlab.rd200p2`) через
  `adb bugreport`, распарсен [`scripts/plus2_btsnoop_parse.py`](../scripts/plus2_btsnoop_parse.py).
  Дал handshake-последовательность и **развязал канал истории + day/month/year-средние**
  (см. §6A). Опкоды `0x54/0x56/0x60/0x61` — это **наблюдённый трафик приложения**, не
  догадки → безопасны.

## 3. Полная GATT-карта (live dump)

```
SERVICE  h=0x0001  1800  Generic Access Profile
  CHAR   h=0x0002  2a00  [read,write]  Device Name
  CHAR   h=0x0004  2a01  [read]        Appearance
  CHAR   h=0x0006  2a04  [read]        Peripheral Preferred Connection Parameters
SERVICE  h=0x0008  1801  Generic Attribute Profile
SERVICE  h=0x0009  00001523-1212-efde-1523-785feabcd123   (Nordic-base, "LED Button Service")
  CHAR   h=0x000a  00001524-...   [read,write]   <- WRITE (команды/опкоды)
  CHAR   h=0x000c  00001525-...   [read,notify]  <- NOTIFY основной (отдаёт кадры)
    DESC h=0x000e  2902  Client Characteristic Configuration (CCCD)
  CHAR   h=0x000f  00001526-...   [read,notify]  <- NOTIFY вторичный (молчал на 0x50/0x51)
    DESC h=0x0011  2902  Client Characteristic Configuration (CCCD)
SERVICE  h=0x0012  180a  Device Information
  CHAR   h=0x0013  2a29  [read]        Manufacturer Name String
```

Полные UUID характеристик service 1523 (Nordic-base
`xxxxxxxx-1212-efde-1523-785feabcd123`):

| Роль | UUID | Handle | Properties | CCCD |
|---|---|---|---|---|
| WRITE | `00001524-1212-efde-1523-785feabcd123` | `0x000a` | read, **write** (with-response) | — |
| NOTIFY-1 | `00001525-1212-efde-1523-785feabcd123` | `0x000c` | read, **notify** | `0x000e` |
| NOTIFY-2 | `00001526-1212-efde-1523-785feabcd123` | `0x000f` | read, **notify** | `0x0011` |

**Наблюдения**:
- Команды пишутся в **1524** одним байтом-опкодом, `write` **with response**
  (свойство `write`, не `write-without-response`).
- Ответы приходят notify на **1525**. Характеристика **1526** существует, имеет
  CCCD, на опкоды 0x50/0x51 **не отдавала** кадры — **РАЗВЯЗАНО (§6A)**: это **канал
  bulk-истории**, активируется только после `0x60`/`0x61`; именно по нему официальное
  приложение качает почасовые записи (handle 0x0010 = value-handle 1526).
- **Важно про handle'ы (§6A)**: ATT-операции идут по **value**-handle (декларация + 1):
  WRITE 1524 → **0x000b**, NOTIFY-1 1525 → **0x000d**, NOTIFY-2 1526 → **0x0010**.

## 4. Опкод-whitelist (HARD — НЕ НАРУШАТЬ)

Разрешённые опкоды: **`{0x10, 0x50, 0x51, 0x53, 0x54, 0x56, 0x60, 0x61, 0xA4, 0xA6, 0xA8, 0xAF, 0xE8, 0xE9}`**.

> ⚠️ **DFU-RISK**: диапазон `0xA0..0xCF` вне whitelist при probe **переводит прибор
> в DFU mode** (инцидент 2026-06-07). Восстановление — вынуть батарейки ~30 с.
> **Никогда** не перебирать опкоды вне whitelist.

> 🛑 **0x53 = ОПАСНАЯ ЗАПИСЬ КОНФИГА — НЕ слать в опросе (INC-RADON-UNIT, 2026-06-16)**:
> `0x53` есть в whitelist в смысле «прибор его распознаёт», но это **config-WRITE**, а НЕ read.
> Голый `0x53` (один байт, без валидного payload) **сбрасывает конфиг прибора в нули** —
> доказано по кадру `0x54` до/после: байт[2] `01→00` (флаг единиц), байт[5] `C8→00`
> (порог тревоги 200→0 Bq/m³), байт[7] `06→00`. Физический эффект на приборе:
> **дисплей переключается Bq/m³ → pCi/L, порог тревоги обнуляется**. Восстановление —
> ТОЛЬКО вручную через приложение RadonEye (единицы + порог). `0x50`/`0x51`/`0x54`/`0x56`
> доказанно НЕ разрушающие (конфиг стабилен через десятки read'ов `0x54`, см. лог 23:02–23:17).
> **УРОК: «whitelist» = опкоды, которые прибор РАСПОЗНАЁТ, а не «безопасно слать вслепую».**
> Операционный опрос шлюза = **ТОЛЬКО `{0x50, 0x51}`**. Любой иной опкод считать потенциально
> пишущим и не слать без точного знания payload.

**Опкоды, подтверждённые трассой официального приложения** (btsnoop 2026-06-15, §6A —
наблюдённый трафик, не догадки → безопасны):

| Опкод | Назначение | Канал ответа |
|---|---|---|
| `0x53` | **config-WRITE** (запись настроек, см. §6B) | echo через `0x54` NOTIFY-1 (0x000d) |
| `0x54` | версия прошивки + конфиг (read) | NOTIFY-1 (0x000d) |

Формат команды: 20 байт = `<opcode> 0x11 <params...>` (`byte[1]` всегда `0x11`).

трассы приложения, не probe'ились вслепую.

## 5. Кадр 0x50 — мгновенный радон (ПОДТВЕРЖДЁН live)

Запрос: записать байт `0x50` в 1524 → ответ notify на 1525, **20 байт**.

Сырой кадр (live, радон ≈ 18 Bq/m³, стабильно 6/6 чтений):
```
сессия 1: 50 10 12 00 00 00 00 00 00 00 00 00 16 00 01 00 02 00 1f 16   (радон 18)
probe-3 : 50 10 22 00 00 00 00 00 00 00 00 00 22 00 00 00 04 00 39 22   (радон 34)
```

Layout:

| Offset | Поле | Тип | Значение (live) | Уверенность |
|---|---|---|---|---|
| `[0]` | opcode echo | u8 | `0x50` | подтверждено |
| `[1]` | длина payload | u8 | `0x10` = 16 | подтверждено |
| `[2:4]` | **радон current** | **uint16 LE Bq/m³** | **18** (= 0.486 pCi/L) | **ПОДТВЕРЖДЕНО** (стабильно 6/6; float32 даёт денормал ≈0) |
| `[4:6]` | **1d average** | uint16 LE Bq/m³ | **336** (2026-06-20) | **ПОДТВЕРЖДЕНО дисплеем 2026-06-20** (захват `50 01` LE = 0x0150 = 336 при дисплее «1d: 336»). Поле пусто в первые сутки работы прибора → отсюда прежние нули в сессиях 2026-06-15. |
| `[6:8]` | **2d average** | uint16 LE Bq/m³ | **321** (2026-06-20) | **ПОДТВЕРЖДЕНО дисплеем 2026-06-20** (захват `41 01` LE = 0x0141 = 321 при дисплее «2d: 321»). Поле пусто в первые двое суток. |
| `[8:12]` | резерв/нули | — | 0 | наблюдение |
| `[12:14]` | **Peak (макс радон)** | uint16 LE Bq/m³ | live=18 (радон=10) | **ПОДТВЕРЖДЕНО дисплеем** (фото RD200P2: «Peak:18» = кадр `[12:14]`=0x12=18 при текущем `[2:4]`=10 → НЕ эхо радона; гипотеза «отслеживает радон» ОПРОВЕРГНУТА: peak≥current во всех образцах: 22≥18, 34≥34, 18≥10) |
| `[14:16]` | **Count (номер замера)** | uint16 LE | live 1→2 | **ПОДТВЕРЖДЕНО дисплеем** (фото: «Count:2» = кадр `[14:16]`; растёт 1→2 в окне) |
| `[16:18]` | вторичный счётчик | uint16 LE | live=2 | дисплей «Count:X/Y» — второе число (`/0` на фото); назначение уточняется |
| `[18:20]` | **часы mm:ss (в 0x50 — КЭШ)** | 2×u8 | `[18]`=мин, `[19]`=сек; 0x50 отдаёт значение с момента последнего 0x51 (не live) — см. §6 | подтверждено |

**КЛЮЧЕВОЙ ВЫВОД (anti-hallucination, доказано live)**: радон current — это
**uint16 LE в Bq/m³**, НЕ float32 pCi/L. Доказательство:
1. Байты `12 00 00 00` как float32 → денормализованный ≈0 (реальные radon-float'ы
   так не выглядят: у них ненулевые байты экспоненты в позициях 2-3).
2. Как uint16 → 18 — осмысленное низкое значение радона.
3. Перекрёстная сверка с образцом EtoTen `50 0a 02 00…` (= 0.05 pCi/L) → uint16
   `[2:4]` = 2 Bq/m³ = 0.054 pCi/L ≈ 0.05 ✓.
4. Стабильность 6/6 чтений (18, 18, 18, 18, 18, 18) при float32-нуле.

**ПОПРАВКА 2026-06-20 (anti-hallucination, новые данные пересматривают вывод 2026-06-15)**:
прежняя «day/month-средние» разметка `[4:6]`/`[6:8]` была снята 2026-06-15 на основании
наблюдения «годовая история, но в этих полях 0». Свежий захват 2026-06-20 (фото дисплея
«1d: 336 / 2d: 321» + одновременный кадр 0x50 от шлюза `radon-gw-c3`) даёт **байт-точное
совпадение** `[4:6]=336`, `[6:8]=321` ↔ дисплей `1d:336 / 2d:321`. Это означает: **поля
есть `1d` и `2d` средние** (НЕ day/month — окно ровно 24 ч и 48 ч, не календарные сутки/месяц).
Старое наблюдение «годовая история, поля = 0» теперь объясняется иначе: прибор тогда
имел uptime ≈ 90-140 мин (виден в `0x51 [4:8]`) — **меньше 24 ч** → 1d ещё не накопилась,
2d тоже. После >2 суток непрерывной работы поля заполняются. Окончательная разметка:
`[4:6]` = **1d** (24-часовое скользящее среднее), `[6:8]` = **2d** (48-часовое скользящее).
Probe 2026-06-15 опровергнутых whitelist-опкодов (`0x10`/`0xE8`/`0xE9` → 0 кадров) остаётся
в силе — отдельных опкодов для day/month нет, агрегаты приходят прямо в `0x50`.

> **РАЗВЯЗАНО (§6A, btsnoop официального приложения 2026-06-15)**: day/month/year-средние
> приложение **не получает отдельным опкодом** и **не обязательно тянет из облака**. Оно
> скачивает почасовую сырую историю по BLE (`0x60` = число записей, `0x61` = bulk-дамп
> через NOTIFY-2/0x0010, 8-байтовые записи `<u32 ts LE><u16 Bq LE><u16 extra>`) и
> **агрегирует средние локально (гипотеза A)**. Прежняя «ведущая гипотеза облако Ecosense»
> — **снята**. Подробности и формат записи — в §6A.

Поля pulse (`[14:16]`/`[16:18]`) — размечены по аналогии с V2, наблюдается рост в
сессии (1→3, 2→5), согласуется со счётчиком импульсов; требует отдельного подтверждения.

## 6. Кадр 0x51 — статус/uptime (декодирован live multi-sample)

Запрос: записать байт `0x51` в 1524 → ответ notify на 1525, **20 байт**.

Разметка получена из 7× 0x51 в окне 110 с (`plus2_0x51_probe.py`, паузы 12/24 с) —
byte-diff анализ показал, какие offset'ы инкрементируют:

| Offset | Поле | Тип | Значение (live) | Уверенность |
|---|---|---|---|---|
| `[0]` | opcode echo | u8 | `0x51` | подтверждено |
| `[1]` | длина payload | u8 | `0x12` = 18 | подтверждено |
| `[2:4]` | статус | uint16 LE | `2` (константа) | наблюдение |
| `[4:8]` | **uptime минуты** | uint32 LE | внутри окна `117 → 119` (+1/мин); кросс-сессия `92 → 117 → 140` | **подтверждено** (тик +1/мин; кросс-сессия задаёт хронологию ~23-25 мин между замерами) |
| `[8:12]` | **неизв. (НЕ счётчик)** | uint32 LE | `527100 → 528004 → 525864` — **немонотонный** | опровергнуто (счётчик не убывает; см. ниже) |
| `[12]` | **Humidity %** | u8 (старший бит — флаг) | **46** (2026-06-20, `0xAE & 0x7F`) | **ПОДТВЕРЖДЕНО дисплеем 2026-06-20** (формула `H = byte[12] & 0x7F`; согласована со всеми старыми наблюдениями: `0xA9→41`, `0xA8→40`, `0xA7→39` ↔ дисплей ≈ 42% ±разброс). Гипотеза `× 0.25` ОПРОВЕРГНУТА. |
| `[13]` | **Temperature °C** | u8 int | **24** (дисплей 24.5; 2026-06-20) | **ПОДТВЕРЖДЕНО** для целой части (24 ↔ дисплей 24.5; дробная часть либо в `[14]`/`[15]`, либо отсутствует — TBD). Старая аналогия Q8.8 не подходит. |
| `[14:18]` | возможно дробь T или fw-метка | 4×u8 | 2026-06-15 `1a 06 0f 01`, 2026-06-20 `1a 06 14 0a` | переразметить (`[14:15]` могут нести дробь T; `[16:17]` варьируются) |
| `[18:20]` | **часы mm:ss (live)** | 2×u8 | `[18]`=мин, `[19]`=сек | **подтверждено по wall-clock** |

**ДОКАЗАТЕЛЬСТВО `[18:20]` = live mm:ss часы (anti-hallucination)**: хвост по
7 кадрам менялся `37 2d → 37 39 → 38 15 → 38 21 → 38 3a → 39 0a → 39 22`, т.е.
`55:45 → 55:57 → 56:21 → 56:33 → 56:58 → 57:10 → 57:34`. Дельты **точно совпадают
с реальными интервалами записи** (12 с / 24 с между write'ами): 55:45→55:57 = +12 с,
55:57→56:21 = +24 с, и т.д. `[18]` = минуты, `[19]` = секунды, **простой hex** (не BCD).

**ПОПРАВКА прежней гипотезы (0x50 кэширует часы)**: в §5 хвост `[18:20]` кадра 0x50
выглядел «session-constant». Probe это **опроверг**: 0x50 отдаёт `[18:20]` со
**значением на момент последнего 0x51** (кэш), а не live. В пробе 0x50-кадры дали
хвосты `{37 39, 38 21, 39 0a}` — это в точности значения предшествующих им 0x51, не
свои. Поэтому в сессии 1 (где все 6 чтений были 0x50) хвост казался «застывшим».
Реальные часы тикают только в ответах 0x51.

Сырые кадры (две сессии, для сверки):
```
сессия 1: 51 12 02 00 5c 00 00 00 fc 0a 08 00 aa 19 1a 06 0f 01 1f 16
probe-2 : 51 12 02 00 75 00 00 00 84 0e 08 00 a8 1a 1a 06 0f 01 37 2d
probe-3 : 51 12 02 00 8c 00 00 00 28 06 08 00 a7 1a 1a 06 0f 02 12 2b
```
Разбор трёх кадров по полям: `[4:8]` uptime `5c→75→8c` = `92→117→140` мин (монотонно,
+25/+23); `[8:12]` `080afc→080e84→080628` = `527100→528004→525864` (немонотонно — НЕ
счётчик); `[17]` `01→01→02` (медленный флаг); `[18:20]` live `1f16→372d→122b` =
`31:22→55:45→18:43` (часы тикают; интервалы совпадают с `[4:8]`).

**СВЕРКА `[8:12]` — «минутный счётчик / наработка» ОКОНЧАТЕЛЬНО ОПРОВЕРГНУТА тремя
сессиями (anti-hallucination, 2026-06-15)**: три хронологических 0x51-замера дали
`[8:12]` = `527100 → 528004 → 525864` (hex `080afc → 080e84 → 080628`).
**Третий замер МЕНЬШЕ второго** (`525864 < 528004`, Δ=−2140) → ряд **немонотонный** →
это **НЕ счётчик** (ни минутный, ни наработки, ни длины истории — счётчик не может
убывать).
**Решающая поправка хронологии (опровергает прежнее «≈15 ч врозь»)**: реальный интервал
между сессиями виден по двум независимым live-полям того же кадра — uptime `[4:8]` и
часам `[18:20]`:
- `[4:8]` (uptime, тикает +1/мин — доказано в окне пробы): `92 → 117 → 140` мин
  → дельты **+25 / +23 мин**.
- `[18:20]` (live mm:ss): `31:22 → 55:45 → 18:43` → +24:23, затем +22:58 (через wrap 60).
Оба поля согласованно показывают, что сессии шли с интервалом **~23-25 минут**, НЕ 15 ч.
Прежнее «15 ч» было допущением, подогнанным под минутный тик (+904 ≈ +900) — фальшивая
корреляция: за реальные ~25 мин минутный счётчик дал бы +25, а не +904.
**Текущая формулировка (HARD)**: `[8:12]` = **неизвестно**. Доказательно НЕ счётчик
(немонотонный ряд `527100 → 528004 → 525864`). Возможные кандидаты — хэш/CRC части
кадра, упакованное многополье или энтропийное значение; для разметки нужен reverse APK
или btsnoop-трасса. Все прежние интерпретации («длина истории», «минуты наработки»,
«lifetime ≈ 1 год») сняты.
**Побочно**: `[17]` (внутри `[14:18]`) по трём сессиям `01 → 01 → 02` — медленный
счётчик/флаг, инкремент ~раз в ~30-48 мин; назначение неизвестно.

**Открыто (низкая уверенность)**: `[8:12]` (немонотонно — не счётчик, см. выше),
`[12:14]` (`aa19/a81a/a71a` — менялось между сессиями), `[14:18]`=`1a 06 0f XX`
(`[14:17]` фикс. — версия прошивки или дата завода; `[17]` медленный флаг `01/01/02`).
Уточнение — reverse APK или btsnoop-трасса.
Где Plus2 отдаёт day/month-средние — **больше НЕ через `0x10`/`0xE8`/`0xE9`**: тщательный
probe 2026-06-15 (`plus2_10e8e9_probe.py`, по 3 записи каждого опкода, окна 10 с) дал
**0 кадров** по всем трём, notify-1526 молчит. Ведущая гипотеза теперь — **облако
Ecosense**: прибор имеет собственный WiFi и шлёт показания в облако напрямую (подтверждено
наблюдением), day/month/year-средние, вероятно, считаются и хранятся в облаке, а приложение
тянет их по HTTPS, **не по BLE**. Это объясняет и `0x50 [4:6]/[6:8]=0`, и тишину
`0x10/0xE8/0xE9`. Развязка — Android HCI-snoop трассы официального приложения (см. §9).

## 6A. Канал истории + day/month/year средние — РАЗВЯЗАНО (btsnoop официального приложения, 2026-06-15)

> **ИТОГ (HARD, доказано трассой официального приложения)**: day/month/year-средние Plus2
> НЕ приходят по BLE как готовые агрегаты и НЕ обязательно тянутся из облака. Официальное
> приложение **скачивает почасовую сырую историю по BLE** (опкод `0x60` = число записей,
> `0x61` = bulk-дамп через NOTIFY-2) и **считает средние на стороне клиента**. Это
> **гипотеза A** (всё выводимо из BLE, облако для средних не нужно). Гипотеза B (облако
> Ecosense) — снята как обязательная: облако существует (у прибора свой WiFi), но средние
> приложение получает локально из BLE-истории.

### Provenance (как снято)

- **Метод**: Android HCI Bluetooth snoop (btsnoop_hci) официального приложения RadonEye+²
  (`kr.ftlab.rd200p2`), выгружено через `adb bugreport` (HyperOS root-gated каталог
  `/data/misc/bluetooth/logs/` — берётся только bug-report'ом).
- **Парсер**: [`scripts/plus2_btsnoop_parse.py`](../scripts/plus2_btsnoop_parse.py) —
  реассемблирует ACL-фрагменты по handle, фильтрует L2CAP CID 0x0004 (ATT), извлекает
  WRITE (0x52/0x12), NOTIFY (0x1b), INDICATE (0x1d) с timestamp/направлением(TX/RX)/
  ATT-handle/value-hex. Применим к любому btsnoop_hci.log.
- **Дата трассы**: 2026-06-15. Прибор реальный (live).
- **Важно (anti-hallucination)**: `0x54/0x56/0x60/0x61` — это **наблюдённый трафик
  официального приложения**, не наши догадки → безопасны для репликации на ESP32.

### Уточнение GATT — ATT работает по VALUE-handle (декларация + 1)

btsnoop показывает ATT-операции на **value**-handle характеристик (декларация в §3 — это
char-declaration handle; value-handle = declaration + 1):

| Роль | char-decl handle (§3) | **VALUE handle (ATT)** | CCCD |
|---|---|---|---|
| WRITE (команды) `1524` | 0x000a | **0x000b** | — |
| NOTIFY-1 (ответы) `1525` | 0x000c | **0x000d** | 0x000e |
| NOTIFY-2 (история) `1526` | 0x000f | **0x0010** | 0x0011 |

Команды пишутся в **0x000b**, ответы-кадры приходят на **0x000d**, **bulk-история — на
0x0010**. (Прежний grep ответов по 0x000c был пуст именно потому, что value-handle =
0x000d.)

### Формат команды (ПОДТВЕРЖДЁН трассой)

20-байтовая запись в handle `0x000b`: `<opcode> 11 <params...>`. **`byte[1]` всегда
`0x11`** во всех командах приложения (0x50/0x51/0x54/0x56/0x60/0x61).

### Handshake официального приложения (verbatim, ответы на 0x000d)

| Команда (write 0x000b) | Ответ (notify 0x000d) | Расшифровка |
|---|---|---|
| `56 11 …` | `56 11 <17 байт ASCII серийника> 00` | серийник/модель ASCII, формат «ГГГГММДД»+«SN»+«####»+«PD2» (значение замаскировано) |
| `54 11 …` | `54 0f 01 00 00 c8 00 01 56 31 2e 30 2e 32 00 01 00 44 32 00` | версия прошивки «V1.0.2» + конфиг (`c8 00`=200?) |
| `50 11 …` | `50 10 0e 00 00 00 00 00 …` | мгновенный радон `[2:4]`=0x000e=14 Bq/m³ (`[4:6]/[6:8]`=0) |
| `51 11 …` | `51 12 02 00 9e 02 00 00 …` | uptime `[4:8]`=0x029e=670 мин |
| `60 11 …` | `60 02 80 25 …` | **число записей истории** `[2:4]`=0x2580 LE=**9600** |
| `61 11 19 00 …` | (поток 8-байтовых записей на **0x0010**) | **bulk-дамп истории**, параметр `[2:4]`=0x0019 LE=**25** записей |

### Протокол истории (РАЗОБРАН)

1. **`0x60`** — запрос числа доступных записей. Ответ на 0x000d: `[2:4]` uint16 LE =
   общее число записей. В трассе = **9600** (≈ 400 дней ≈ 13 месяцев почасовой истории).
2. **`0x61`** — bulk-дамп. Формат команды `61 11 <N_u16_LE> 00…`. Параметр `[2:4]` uint16 LE =
   **сколько записей выдать**, причём это **N САМЫХ СВЕЖИХ почасовых записей**
   (доказано dump #0: req=25 → rec0=2026-06-14 16:22, rec24=2026-06-15 19:26 = ровно
   последние 25 часов). N=9600 = вся история, N=240 = последние 10 дней, N=25 = последние 25 ч.
   Burst ровно `N × 8` байт (25→200 B — точное совпадение, доказывает «param = число записей»).
3. Записи приходят непрерывным байтовым потоком на NOTIFY-2 (**handle 0x0010**), нарезаются
   по 8 байт, порядок oldest→newest.

**«Скачать историю за любой период» — РЕШЕНО двумя путями:**
- **(a)** запросить `61 11 <N_LE> 00`, где N = число часов назад от текущего момента
  (N свежих почасовых записей);
- **(b)** запросить полный дамп (N=9600) и отфильтровать по встроенному таймстампу `[0:4]`
  под произвольное окно `[start, end]`. Оба способа проверены на трассе.

### Формат записи истории (8 байт, ДЕКОДИРОВАН)

| Offset | Поле | Тип | Уверенность |
|---|---|---|---|
| `[0:4]` | **Unix-таймстамп** | uint32 LE, секунды, **локальное время прибора** | **подтверждён** (шаг +3600 с = почасовые записи; последняя запись ≈ момент захвата трассы; порядок oldest→newest) |
| `[4:6]` | **радон** | uint16 LE Bq/m³ | **подтверждён** (полный дамп 9600 записей: min=0 / max=884 / mean=132.3 — реалистичный приподнятый подвал; та же единица, что и 0x50 instant) |
| `[6:8]` | **температура** | uint16 LE = °C × 256 (Q8.8, делить на 256) | **подтверждён** (по 9600 записям: min=20.2 / max=27.6 / mean=23.6 °C; `[7]`=целые °C 20-27, `[6]`=дробная часть; физически согласовано — май ≈23 °C, июнь ≈25-26 °C) |

Декодирование (полный дамп 9600 записей из btsnoop, oldest→newest):
- rec 0: `4fc81e681c009717` → ts=2025-05-10 03:30 (локальн.), радон=28 Bq/m³, темп=0x17/256·256+0x97/256 ≈ 23.6 °C
- rec 9599: `e651306a0a002d19` → ts=2026-06-15 19:26 (локальн.) ≈ момент захвата, радон=10 Bq/m³, темп ≈ 25.2 °C
- span = **401.7 дней**, шаг строго +3600 с, **75/9599 разрывов ≠ 3600 с** = события выключения прибора за >1 год (норма, НЕ баг).

### Вывод по day/month/year

Приложение **не запрашивает** отдельный опкод «среднее за день/месяц/год». Оно тянет
почасовую сырую историю (`0x60`+`0x61`) и **агрегирует средние локально**. Это объясняет
`0x50 [4:6]/[6:8]=0` (там нет агрегатов) и тишину `0x10/0xE8/0xE9` (они не используются
приложением вовсе). Канал NOTIFY-2 (`1526`/0x0010), молчавший на всех прямых probe'ах, —
это **канал истории**: активируется только после `0x60`/`0x61`.

### Рекомендация для ESP32-шлюза

day/month/year **выводимы из BLE** без облака. Два пути:
- **(A) Реплицировать `0x60`+`0x61`** на шлюзе: скачать почасовую историю, нарезать по
  8 байт, агрегировать средние (день=24 записи, месяц≈720, год≈8760). Точно как приложение.
- **(B) Считать на лету** из live-потока `0x50` (наш текущий ring-buffer подход) —
  но это повторяет heap-риск Radex (json:111). Учитывая урок Radex — историю лучше
  отдавать сырой в HA Recorder и агрегировать там, либо ограничить буфер.
- Прагматично: на стартовом шлюзе — `0x50` мгновенный + (опц.) разовый `0x60`/`0x61`-дамп
  истории при подключении для бэкафилла, агрегаты — на стороне HA.

## 6B. Кадр 0x53 — config-WRITE (запись настроек, РАЗОБРАН частично)

Опкод записи настроек прибора (трасса официального приложения, btsnoop 2026-06-15). Пишется
в WRITE-handle 0x000b, прибор подтверждает **echo через `0x54` NOTIFY** на 0x000d.

Наблюдённые кадры (20 байт):
```
53 11 01 00 00 c8 00 01 00 00 00 00 00 00 00 00 00 00 00 00
53 11 01 00 00 c8 00 06 00 00 00 00 00 00 00 00 00 00 00 00
```

| Offset | Значение | Назначение (гипотеза) |
|---|---|---|
| `[0]` | `0x53` | опкод config-WRITE |
| `[1]` | `0x11` | константа команды |
| `[2]` | `0x01` | подкоманда / тип записи |
| `[3:5]` | `00 00` | reserved |
| `[5:7]` | `c8 00` = 200 LE | **порог тревоги (alarm threshold)** — совпадает с `0x54` config blob |
| `[7]` | `0x01` / `0x06` | **код изменяемого поля** (варьируется между кадрами) |
| `[8:20]` | нули | padding |

`0x54` config blob (read-ответ): `54 0f 01 00 00 | c8 00 | XX | "V1.0.2\0" | 01 00` —
`c8 00`=200 = тот же порог тревоги.

> ⚠️ **Изменение настроек на реальном приборе пользователя** — порог тревоги и пр. менять
> только по явному запросу. Для проверки записи предпочитать безобидное обратимое поле и
> восстанавливать исходное значение. История (`0x60`/`0x61`) — READ-ONLY, безопасна.

Полная байтовая карта `0x53` (единица измерения / вкл-выкл тревоги / интервал / °C-°F /
зуммер) — TODO (§9).

## 7. Конверсия единиц

- **1 pCi/L = 37 Bq/m³** (точно).
- Bq/m³ → pCi/L: делить на 37. pCi/L → Bq/m³: умножать на 37.
- Прибор отдаёт **Bq/m³** (uint16). Пример: 18 Bq/m³ / 37 = 0.486 pCi/L.

## 8. Минимальный рецепт чтения (любой стек)

1. Scan → найти устройство по adv-префиксу `FR:PD` (или по известному MAC).
2. Connect (single central; прибор не держит >1 коннекта — освободить ESP-мост,
   если он подключён).
3. Subscribe notify на `00001525-...` (CCCD `0x000e`).
4. Write `0x50` (1 байт, with-response) в `00001524-...` (handle `0x000a`).
5. Принять 20-байтовый notify-кадр на 1525.
6. `radon_Bq = uint16_LE(frame[2:4])`; `radon_pCi = radon_Bq / 37`.
7. (Опц.) `0x51` → статус/uptime-кадр.

ESPHome `ble_client`: write в char `00001524-...` (`type: write`),
sensor `type: characteristic` notify на `00001525-...`, в лямбде
`x[2] | (x[3] << 8)` → Bq/m³.
